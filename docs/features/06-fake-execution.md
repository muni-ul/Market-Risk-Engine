# Feature 6 — Fake Execution

## 1. Feature Overview

### Feature name

Fake Execution

### One-sentence description

Convert strategy-generated buy/sell signals into deterministic simulated orders and fake trade fills using the option prices produced by the pricing engine.

### Detailed description

The Fake Execution feature is the bridge between the Simple Fake Strategy and the later Portfolio Tracker. It takes signal records, converts actionable signals into order objects, fills those orders at deterministic simulated prices, and exports clean trade logs for later portfolio accounting.

This feature is intentionally fake and local-only. It does not connect to a brokerage, does not submit real orders, does not model a real exchange, and does not claim trading profitability. Its purpose is to create realistic-looking trade records from the existing simulated pricing data so the rest of the PyRiskLab pipeline can demonstrate state management, reporting, testing, and reproducibility.

The execution model should be simple:

1. Read strategy signals from the in-memory signal list or `signals.csv`.
2. Ignore `hold` signals.
3. Convert `buy` and `sell` signals into order records.
4. Match each order to the option price at the same simulation step.
5. Fill the order deterministically at that price.
6. Save `orders.csv` and `trades.csv` into `results/<run_name>/`.
7. Return trade objects for the Portfolio Tracker.

### Why it matters

This feature turns the project from a pricing/reporting tool into a small simulation system with an action layer. It creates the trade records needed for portfolio accounting, drawdown analysis, risk rules, and final reporting.

It also gives reviewers evidence that the project has a clean internal pipeline:

```text
pricing_history -> greeks_history -> signals -> orders -> trades -> portfolio_history
```

### Skill it demonstrates

- Dataclass-based domain modeling.
- Deterministic simulation design.
- Clean separation of strategy, execution, portfolio, and risk logic.
- Input validation.
- pandas export pipelines.
- Testable state transitions.
- Local file-based workflow.
- Clear boundaries around fake execution versus real trading.

### Priority

High.

Justification: The portfolio tracker cannot be meaningful without trade records. This feature is also a major step toward making PyRiskLab feel like a complete engineering system instead of only a formula calculator.

### Complexity

Medium.

Justification: The logic is not mathematically hard, but it needs careful boundaries. It must generate realistic records without accidentally becoming a real trading system, order-book simulator, brokerage integration, or risk manager.

## 2. User/Demo Flow

### Happy path

1. User runs:

```bash
python -m pyrisklab run --config configs/demo.yaml
```

2. Earlier pipeline steps generate:
   - `market_path.csv`
   - `pricing_history.csv`
   - `greeks_history.csv`
   - `signals.csv`

3. Fake Execution receives the generated signals and pricing history.
4. `hold` signals are ignored.
5. `buy` and `sell` signals become orders.
6. Each order is filled at the option price from the same step.
7. The app saves:
   - `orders.csv`
   - `trades.csv`
8. CLI prints a concise message:

```text
[6/12] Executing fake trades...
Created 14 orders
Filled 14 simulated trades
Saved orders.csv
Saved trades.csv
```

### First-time path

A first-time reviewer should not need to understand trading mechanics. The README and summary report should explain that fake execution means:

> Strategy signals are converted into simulated trades using prices generated by the model. No real money, market data, brokerage, or live trading is involved.

The output tables should be readable without opening the code.

### Empty state

If the strategy generates no actionable `buy` or `sell` signals:

- `orders.csv` is still created with headers and zero rows.
- `trades.csv` is still created with headers and zero rows.
- CLI prints:

```text
No actionable signals found. Created empty orders.csv and trades.csv.
```

- The later Portfolio Tracker should still be able to run with no trades.

### Error path

Examples of expected errors:

- Signal action is not one of `buy`, `sell`, or `hold`.
- Signal quantity is missing or invalid.
- Signal step does not exist in `pricing_history`.
- Fill price is missing, negative, `NaN`, or not finite.
- Order side is invalid.
- Order quantity is less than or equal to zero.
- Required pricing columns are missing.

Errors should be clear and actionable:

```text
ExecutionError: signal at step 44 has action 'BUY_NOW'. Expected one of: buy, sell, hold.
```

```text
ExecutionError: cannot fill order at step 91 because pricing_history has no option_price for that step.
```

### Demo path for a reviewer

Reviewer runs:

```bash
python -m pyrisklab run --config configs/demo.yaml --overwrite
```

Then opens:

```text
results/demo_run/orders.csv
results/demo_run/trades.csv
```

They should see:

- Each actionable signal became an order.
- Each order became a fake trade.
- Fill prices came from simulated option prices.
- The records include enough fields for portfolio accounting.
- No real trading or brokerage integration exists.

## 5. Logic Requirements

### Business rules

1. Only `buy` and `sell` signals are actionable.
2. `hold` signals produce no order.
3. Every actionable signal creates exactly one order.
4. Every valid order is filled immediately in MVP.
5. Fill price equals `pricing_history.option_price` for the same `step` and `symbol`.
6. Order type should be `market` for MVP.
7. Fill model should be `deterministic_mid` for MVP.
8. Notional is calculated as:

```text
notional = fill_price * quantity * contract_multiplier
```

9. Commission is calculated as:

```text
commission = commission_per_contract * quantity
```

10. Trade output should include both notional and commission.
11. Trade IDs and order IDs should be deterministic for reproducibility.
12. The same inputs must produce the same `orders.csv` and `trades.csv`.

### Calculations

#### Requested price

```text
requested_price = option_price at matching step and symbol
```

#### Fill price

```text
fill_price = requested_price
```

#### Notional

```text
notional = fill_price * quantity * contract_multiplier
```

#### Commission

```text
commission = commission_per_contract * quantity
```

#### Signed notional

Optional for output, useful for debugging:

```text
signed_notional = -notional for buy
signed_notional = +notional for sell
```

[Open Question] Should `signed_notional` be included in `trades.csv`?

Recommended decision: include it if it simplifies Portfolio Tracker implementation; otherwise keep `side` + `notional` only.

### API/service functions if needed

Recommended functions in `execution.py`:

```python
def create_orders_from_signals(
    signals: pd.DataFrame,
    pricing_history: pd.DataFrame,
    *,
    default_order_type: str = "market",
) -> pd.DataFrame:
    """Convert buy/sell signals into deterministic order records."""
```

```python
def execute_orders(
    orders: pd.DataFrame,
    *,
    commission_per_contract: float = 0.0,
    contract_multiplier: int = 100,
    fill_model: str = "deterministic_mid",
) -> pd.DataFrame:
    """Convert valid fake orders into fake filled trades."""
```

Alternative object-oriented version:

```python
class FakeExecutionEngine:
    def create_orders(self, signals: pd.DataFrame, pricing_history: pd.DataFrame) -> pd.DataFrame:
        ...

    def execute_orders(self, orders: pd.DataFrame) -> pd.DataFrame:
        ...
```

Recommended decision: use pure functions first unless Codex already established a clean class pattern.

### State management

Fake Execution should be mostly stateless.

Allowed internal state:

- Sequential deterministic order ID generation inside a run.
- Sequential deterministic trade ID generation inside a run.

Avoid:

- Global counters.
- Module-level mutable state.
- Random fill behavior.
- Hidden config values.
- Portfolio state.
- Risk state.

### Edge cases

Must handle:

- No signals.
- All signals are `hold`.
- Invalid signal action.
- Missing `quantity`.
- Quantity equals zero for `buy` or `sell`.
- Negative quantity.
- Missing matching price.
- Duplicate pricing rows for the same `step` and `symbol`.
- Missing `option_price` column.
- `option_price` is `NaN` or infinite.
- `option_price` is negative.
- Commission is negative.
- Contract multiplier is zero or negative.
- Unsupported fill model.

Recommended duplicate-price behavior:

- Raise `ExecutionError` if there are multiple pricing rows for the same `step` and `symbol`.
- This prevents silent ambiguous fills.

## 6. Acceptance Criteria

### Scenario 1 — Buy signal creates order and trade

Given a signal with action `buy`, quantity `1`, symbol `CALL_105`, and step `10`  
And pricing history contains `CALL_105` at step `10` with option price `4.50`  
When fake execution runs  
Then one order should be created  
And one trade should be created  
And the trade fill price should be `4.50`  
And the trade notional should equal `4.50 * 1 * contract_multiplier`.

### Scenario 2 — Sell signal creates order and trade

Given a signal with action `sell`, quantity `2`, symbol `CALL_105`, and step `20`  
And pricing history contains a matching option price  
When fake execution runs  
Then one sell order should be created  
And one sell trade should be created  
And the trade side should be `sell`.

### Scenario 3 — Hold signal is ignored

Given a signal with action `hold`  
When fake execution runs  
Then no order should be created for that signal  
And no trade should be created for that signal.

### Scenario 4 — No actionable signals still creates empty files

Given all strategy signals are `hold`  
When the pipeline runs  
Then `orders.csv` should be created with headers  
And `trades.csv` should be created with headers  
And the pipeline should continue without crashing.

### Scenario 5 — Missing pricing row fails clearly

Given a buy signal at step `50`  
And pricing history has no row for step `50` and the signal symbol  
When fake execution runs  
Then the run should fail with an `ExecutionError`  
And the error message should mention the missing step and symbol.

### Scenario 6 — Invalid action fails clearly

Given a signal with action `buy_now`  
When fake execution validates signals  
Then the run should fail with an `ExecutionError`  
And the message should state that action must be `buy`, `sell`, or `hold`.

### Scenario 7 — Invalid quantity fails clearly

Given a buy signal with quantity `0`  
When fake execution validates the signal  
Then the run should fail with an `ExecutionError`  
And the message should mention that actionable signal quantity must be greater than zero.

### Scenario 8 — Same inputs produce same output

Given the same signals, pricing history, and execution config  
When fake execution runs twice  
Then the generated order IDs, trade IDs, fill prices, and notional values should be identical.

### Scenario 9 — Unsupported fill model fails clearly

Given execution config uses `fill_model: random_slippage`  
When fake execution runs  
Then the run should fail with an `ExecutionError`  
And the message should explain that only `deterministic_mid` is supported in MVP.

## 8. Portfolio Value

### How this feature helps the project stand out

Fake Execution makes PyRiskLab feel like a complete simulation pipeline instead of a static calculator. It demonstrates that the project can transform model outputs into structured events, which is a useful software engineering pattern.

Recruiters may not care about trade mechanics, but they can understand that the project has:

- Input data.
- Model output.
- Decision signals.
- Simulated actions.
- Logs.
- Later portfolio state.

That is stronger than a notebook that only plots prices.

### What to mention in README

Mention:

- Fake execution is deterministic and simulation-only.
- Strategy signals are converted into orders and trades.
- Trades are filled using generated option prices.
- No real money, brokerage, or live market data is used.
- The output is saved in `orders.csv` and `trades.csv`.

Suggested README wording:

> The fake execution layer converts strategy signals into simulated orders and fills them at the model-generated option price for that step. This creates deterministic trade logs for portfolio accounting while avoiding any real trading or brokerage integration.

### What to mention in interviews

Good interview explanation:

> I separated strategy from execution on purpose. The strategy only says buy, sell, or hold. The fake execution layer turns those signals into deterministic trade records, using the pricing engine's option price as the fill price. I kept this intentionally simple because the goal was not to model a real exchange; it was to create clean, testable state transitions for the portfolio and risk modules.

Potential interviewer questions:

- Why did you separate signals, orders, and trades?
- Why did you avoid real brokerage APIs?
- How do you guarantee reproducibility?
- How do you prevent invalid trades?
- What would you add in a more realistic execution simulator?

Strong answer theme:

> I kept execution deterministic in Version 1 so tests and reports are reproducible. If I wanted more realism later, I would add slippage/spread models, but only after the core accounting and risk logic were correct.
